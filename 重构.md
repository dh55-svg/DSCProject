 MYDSC 项目框架重构方案

 Context

 当前项目 8 个月从零堆到 30+ 个文件、5 级实时管道，功能完整。但软件设计层面存在系统性债务：单例泛滥（13 个）、缺少接口抽象、依赖硬编码、DataEngine
 职责过重、DatabaseManager 12 表一锅端。如果以 8 年+ C++ 工程师标准重构，目标是可扩展、可测试、可维护的工业级架构。

 核心改动

1. 消灭单例 → 依赖注入
   
   现状： 13 个 instance() 散落各处
   // 现状 — 硬编码依赖
   AlarmEngine::instance().triggerAlarm(...);
   AuthManager::instance().canOperate();
   DatabaseManager::instance().insertOperationLog(...);
   
   重构： AppContext 集中持有所有核心对象
   struct AppContext {
    // 领域服务
    std::shared_ptr<AlarmEngine>     alarmEngine;
    std::shared_ptr<AuthManager>     authManager;
    std::shared_ptr<TagManager>      tagManager;
    std::shared_ptr<DataPipeline>    dataPipeline;
   
    // 基础设施（接口注入）
    std::shared_ptr<IFieldbus>       fieldbus;
    std::shared_ptr<IAlarmRepo>      alarmRepo;
    std::shared_ptr<IHistoryRepo>    historyRepo;
    std::shared_ptr<ITagRepo>        tagRepo;
    std::shared_ptr<IUserRepo>       userRepo;
    std::shared_ptr<ILogger>         logger;
   };
   
   // 在 main() 中组装，传入控制器
   auto ctx = ApplicationBuilder()
    .withFieldbus<ModbusImpl>()
    .withDatabase<MysqlImpl>()
    .build();
   DataController dataCtrl(ctx);

2. 接口抽象 → 可替换实现
   
   // 现场总线接口
   class IFieldbus {
   public:
    virtual ~IFieldbus() = default;
    virtual void addDevice(const DeviceConfig& cfg) = 0;
    virtual void startAll() = 0;
    virtual void stopAll() = 0;
    virtual void writeRegister(int devId, int addr, quint16 val) = 0;
    virtual void setDataSink(IMessageBus* sink) = 0;  // 解耦输出
   };

   };
   
   // 协议实现互不感知
   class ModbusImpl    : public IFieldbus { ... };
   class OpcUaImpl     : public IFieldbus { ... };

3. DataEngine 拆分 → 管道独立
   
   现状： DataEngine 负责装配 + Modbus 管理 + 配置加载 + 信号管道 + SP/OUT/模式下发
   
   重构：
   class DataPipeline {   // 纯管道，只关心数据流
    LockFreeRingBuffer<RawSample, 8192>  m_ringBuffer;  // 值对象，解耦 Modbus
    std::unique_ptr<DataParseThread>     m_parser;
    DoubleBuffer                         m_doubleBuffer;
    std::unique_ptr<HistorySampler>      m_sampler;
   public:
    void injectSource(IMessageBus* source);
    void start();
    void stop();
    DoubleBuffer* output() { return &m_doubleBuffer; }
   };
   
   class ModbusController {   // 只管理 Modbus 设备
    ModbusManager m_manager;
   public:
    void addDevice(const DeviceConfig& cfg);
    void startAll();
    void writeSetPoint(int devId, int reg, float val);
   signals:
    void deviceOnline(int id);
    void deviceOffline(int id);
   };
   
   class ConfigLoader {   // 只负责 JSON → TagInfo 转换
   public:
    QVector<TagInfo> loadFromFile(const QString& path);
    void validate(const QVector<TagInfo>& tags);
   };

4. 数据层 → Repository 模式
   
   现状： DatabaseManager 12 张表 SQL 混在一起，1500+ 行
   
   重构： 按聚合根拆分
   IAlarmRepo       → AlarmMysqlRepo / AlarmSqliteRepo    (alarm_events, alarm_change_log, alarm_kpi)
   IHistoryRepo     → HistoryMysqlRepo                     (history_data)
   ITagRepo         → TagMysqlRepo                         (tags)
   IUserRepo        → UserMysqlRepo                        (users, user_roles, user_sessions)
   IOperationRepo   → OperationMysqlRepo                   (operation_log)
   ISystemRepo      → SystemMysqlRepo                      (system_events, shift_logs, backup_log)
   
   每个 Repo 持有自己的 QSqlDatabase 连接名，通过连接池复用。

5. 领域模型独立
   
   现状： TagDef.h 同时放数据结构和 DB 映射逻辑
   
   重构：
   domain/
   ├── TagInfo.h           // 纯领域对象
   ├── AlarmEvent.h        // 报警事件
   ├── AlarmStateMachine.h // 7 状态状态机（纯逻辑，不依赖 Qt）
   ├── AlarmKpi.h          // KPI 计算
   ├── ShelveManager.h     // 搁置管理
   ├── SuppressionEngine.h // 条件抑制规则引擎
   ├── FloodDetector.h     // 泛滥检测
   ├── ChatteringGuard.h   // 抖振保护
   ├── User.h              // 用户实体
   └── DeadbandFilter.h    // 死区滞环

6. 线程管理 → RAII 生命周期
   
   现状： 手动 start()/stop()/wait(3000)，容易泄漏
   
   重构：
   class ThreadGuard {
    QThread m_thread;
    std::atomic<bool> m_running{false};
   public:
    template<typename Worker>
    void launch(Worker&& w) {
   
        m_running = true;
        QObject::connect(&m_thread, &QThread::finished, [&] { m_running = false; });
        // ...
   
    }
    ~ThreadGuard() {
   
        m_running = false;
        m_thread.quit();
        m_thread.wait(5000);
   
    }
   };

7. 配置化启动
   
   // main.cpp — 组装入口
   int main(int argc, char* argv[]) {
    QApplication app(argc, argv);
   
    // 1. 加载配置
    auto appCfg = AppConfig::fromJson("config/app.json");
   
    // 2. 选择后端
    auto db = appCfg.dbBackend == "mysql"
   
        ? std::make_shared<MysqlConnectionPool>(appCfg.mysql)
        : std::make_shared<SqliteConnection>("data/dcs.db");
   
    // 3. 组装依赖图
    AppContext ctx;
    ctx.tagRepo      = std::make_shared<TagMysqlRepo>(db);
    ctx.alarmRepo    = std::make_shared<AlarmMysqlRepo>(db);
    ctx.historyRepo  = std::make_shared<HistoryMysqlRepo>(db);
    ctx.fieldbus     = std::make_shared<ModbusImpl>();
    ctx.alarmEngine  = std::make_shared<AlarmEngine>(*ctx.alarmRepo);
    ctx.authManager  = std::make_shared<AuthManager>(*ctx.userRepo, *ctx.operationRepo);
    ctx.tagManager   = std::make_shared<TagManager>(*ctx.tagRepo);
    ctx.dataPipeline = std::make_shared<DataPipeline>();
    ctx.dataPipeline->injectTagConfig(ctx.tagManager->allTags());
    ctx.dataPipeline->injectSource(ctx.fieldbus.get());
   
    // 4. 控制器
    DataController  dataCtrl(ctx);
    AlarmController alarmCtrl(ctx);
    AuthController  authCtrl(ctx);
   
    // 5. 主窗口
    MYDSCMainWindow window(dataCtrl, alarmCtrl, authCtrl);
    window.show();
   
    return app.exec();
   }

   重构原则

8. 依赖倒置：上层依赖接口，下层实现接口

9. 单一职责：每个类只有一个理由改变

10. 开闭原则：加 OPC UA 不改 PipeLine 内部代码，加新表不改 Domain 层

11. RAII 管理一切资源：线程、连接、定时器生命周期自动

12. 值对象传数据，接口传行为：TagSnapshot/PIDConfig 是值对象，IFieldbus/IAlarmRepo 是接口

13. Qt 信号槽仅用于表现层通信：跨层调用走接口，不走信号
    
    不做什么
- 不引入 Boost、Poco 等重型依赖（保持依赖最小化）
- 不过度模板化（CRTP、Policy-based 等，团队看不懂）
- 不拆过分（一个类 30 行拆成 5 个类的过度工程）
- 不引入消息队列中间件（单机部署不需要）
- 不写测试（按用户要求暂不考虑）
